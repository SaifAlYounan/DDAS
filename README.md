# DDAS — Dynamic Delegation of Authority System

> *Static approval matrices assume a static world. This one doesn't.*

DDAS replaces the corporate **Delegation of Authority table** with your registered **risk appetite, executed as code**. Any transaction or action — initiated by a human or an AI agent — is classified against your own enterprise risk matrix and risk tolerance, producing the required authority tier and a fully auditable derivation. The same action, in a different context, routes to a different approver. Not because the rules changed — because the risk did.

**Self-hosted, open source (MIT), single container + Postgres.** `docker compose up` is the whole product: REST API, web console, approval routing, background workers, and an MCP server for AI agents.

---

## How it decides — no scores, no weights

DDAS uses **Appetite-Constrained Ordinal Sorting (ACOS)**. There is no composite "risk score", no weights, and no LLM assigning numbers. (The reasoning, and why the v1 weighted-sum model was retired, is in [ADR 0004](docs/adr/0004-acos-risk-model.md).)

1. **Extract** — an LLM reads the transaction documents and extracts typed facts (amounts, counterparty, jurisdiction, liability caps, termination rights…). Every fact carries a **verbatim citation** that must string-match the source, and an explicit `NOT_FOUND` when absent. This is the only machine-learning step.
2. **Map** — deterministic band rules place each fact on **your** impact and likelihood scales, per risk category.
3. **Look up** — (impact, likelihood) indexes **your** registered risk matrix → a rating in your vocabulary.
4. **Compare** — **your** registered appetite (rating → minimum authority tier, per category) is the machine-readable risk appetite statement.
5. **Compose, upward only** — final tier = the maximum across categories (one breached ceiling routes the whole action), then escalation triggers, a counts-not-sums accumulation rule, and a stricter appetite for agent-initiated actions. Missing information escalates or blocks; it never classifies as low risk.

Every decision explains itself in your own language, from a **derivation object** generated by templates — never by an LLM — so the explanation cannot diverge from the computation. Replaying the same facts under the same policy version reproduces the derivation byte-for-byte. **That replay is the audit procedure.**

> Where does the risk matrix and appetite come from? You **author them as a policy file** (YAML) and activate it in the Policies console — not by clicking cells in a UI. Your org structure, users, and SLAs *are* configured in the UI. Both steps are walked through below.

## 1. Quickstart (Docker Compose)

**Prerequisites:** Docker with Compose v2 (`docker compose version`). Nothing else — no local Node, Postgres, or build tools needed for this path.

```bash
git clone https://github.com/SaifAlYounan/DDAS.git
cd DDAS/deploy
```

Open `deploy/docker-compose.yml` and set, at minimum, the bootstrap admin (created only on first boot, only if no admin exists yet):

```yaml
    environment:
      DDAS_ADMIN_EMAIL: you@example.com
      DDAS_ADMIN_PASSWORD: choose-a-strong-one   # >= 12 characters
```

Then bring it up:

```bash
docker compose up --build
```

- The app is at **http://localhost:3000**. If port 3000 is taken, run `DDAS_PORT=3210 docker compose up --build` and use that port instead.
- Database migrations apply automatically at boot.
- Log in with the admin email/password you set.

You now have a running DDAS with an empty org and no policy. The next two sections fix that.

> **Trying it without an LLM?** Leave the `DDAS_EXTRACTION_*` variables commented out and set `DDAS_EXTRACTION_PROVIDER: stub` — the stub extractor returns every fact as `NOT_FOUND`, so you enter facts by hand in the review screen. Perfect for a first walkthrough. To wire a real model, see [Extraction providers](#extraction-providers).

## 2. Set up your organisation (in the console)

DDAS routes to **positions in your org**, evaluated **as of the moment of the request**. Configure this under **Admin** and **Org**.

### 2a. People and roles — Admin

Every person is a **principal**. There are six fixed roles; a principal can hold several. Admin implies all of them.

| Role | Can |
|---|---|
| `admin` | everything: users, org, API keys, webhooks, settings |
| `policy_author` | draft/lint/simulate/activate/retire policies |
| `requester` | submit requests, review facts |
| `approver` | see the inbox, approve/reject, review facts |
| `auditor` | read the audit log, verify the chain, replay classifications |
| `viewer` | read-only: browse every request, classification, derivation, policy, and the org tree — no writes, no admin surfaces, no audit-chain endpoints |

In **Admin → Principals → New**, create the people you need. Give at least one person `requester`, and enough people `approver` to cover every authority tier your policy uses. (You can also create **agents** here — an agent must name an accountable human owner; see [AI agents](#6-ai-agents-api-keys--mcp).)

### 2b. Org units, positions, assignments — Org

- **Units** form a tree (e.g. company → Finance → Procurement).
- **Positions** live in a unit and carry an **authority tier** (an integer matching your policy's ladder — see §3). A CFO position might be tier 3.
- **Assignments** put a person in a position for a **validity window**. A vacant position (no live assignment) is simply skipped when routing — *temporal validity is the absence model*, so you model someone being on leave by ending their assignment, not deleting it.
- **Delegations** (optional) let one person approve on another's behalf, up to a tier ceiling and within an org subtree, for a window. Delegations are **one hop** — they never chain.

Routing walks up from the requester's unit until it finds enough live approvers at or above the required tier.

### 2c. Approval SLAs — Admin → Settings

Set the SLA hours per tier. When a task isn't decided in time, DDAS re-resolves one tier higher and **adds** those approvers to the task (the originals stay eligible); it repeats up the ladder.

## 3. Register your risk policy (the risk matrix + appetite)

This is where your **risk matrix and risk appetite** live. A policy is a YAML document; DDAS compiles it to canonical JSON, content-hashes it, and treats it as immutable once active. Start from the annotated template:

```
packages/policy/templates/starter-balanced.yaml
```

It recasts six common risk categories (financial, reversibility, regulatory, reputational, precedent, stakeholder complexity) in ACOS form. Onboarding = replacing its example numbers with what your risk function already maintains. The pieces:

- **`authority_ladder`** — your approval tiers, ascending. Tier 0 is self-approval. Each tier names the roles that hold it and an optional `quorum` (a number, or `majority`). These tier numbers are the same integers you put on positions in §2b.
  ```yaml
  authority_ladder:
    - { tier: 0, name: Self-approve }
    - { tier: 1, name: Manager, roles: [line_manager] }
    - { tier: 2, name: Director, roles: [function_head] }
    - { tier: 3, name: C-Suite, roles: [cfo, general_counsel], quorum: 1 }
    - { tier: 4, name: Board, roles: [board], quorum: majority }
  ```
- **`likelihood_scale`** and **`rating_scale`** — the ordered band names you use (e.g. `Rare … Almost_certain`; `Low … Critical`).
- **`fact_schema`** — every fact the extractor may produce, with its type (`money`, `number`, `string`, `boolean`, `enum`, …). Facts are the only input to classification.
- **`categories`** — the heart of it. Each category declares:
  - **`impact_scale.bands`** — rules that map facts to an impact band on *your* scale (`amount_base_total < 250000 → Moderate`). The last band is `else`. `required_facts` lists what the rules need.
  - **`likelihood_rules`** — how facts (or their absence) set the likelihood band.
  - **`risk_matrix`** — **your grid**: impact band × likelihood band → a rating. This is the risk matrix your CRO already owns.
  - **`appetite`** — **your risk appetite**: rating → the minimum authority tier that may approve it. `appetite_agent_initiated` is an optional stricter column for AI-agent requests.
  - **`missing_info`** — what to do when a required fact is `NOT_FOUND`: `escalate` (assume a conservative band) or `needs_info` (block classification as INCOMPLETE). Never silently low-risk.
- **`escalation_triggers`** — named boolean rules that force a minimum tier or add an uplift (sanctions exposure → Board; uncapped liability → +1 tier).
- **`accumulation_rule`** — "≥ N categories at rating X or above → +1 tier". Counts, never sums.
- **`agent_policy`** — the default agent uplift, whether agents may ever self-approve, and `attestation_required_facts` — facts an agent must have its **human owner** attest (DDAS refuses to let an agent attest these itself).
- **`reference_lists`** — named lists your rules reference (sanctions jurisdictions, approved precedents…).
- **`fx_snapshot`** — the FX rates money facts normalize through, pinned into the policy version so decisions never drift with the market.

### Validate and activate

Lint before you ship — either from the CLI or right in the console:

```bash
node apps/cli/dist/main.js policy lint packages/policy/templates/starter-balanced.yaml
```

In the console: **Policies → New draft**, paste the YAML, and DDAS shows lint findings inline. Error-severity findings block activation; fix them until the list is clean. Then **Activate**.

- The **first** activation of a policy has no history to test against, so DDAS asks for a typed **override reason** (recorded in the audit log).
- Later activations expect a **simulation run** instead: draft the new version, run a **Simulate** against your stored requests ("under this policy, N requests would route differently"), and activate against that run. Activating without a simulation still works, but demands an explicit audited override.

## 4. Your first request, end to end

1. **Submit** — **Requests → New**. Give it a title, pick the active policy, and attach the transaction document(s) (`.txt` or `.md` in this release). Submit.
2. **Extraction** — the request enters *extracting*; the LLM (or the stub) pulls facts with citations. When it finishes it moves to *facts review*.
3. **Review the facts** — the fact-review screen shows every fact with its status and, for extracted ones, the **citation highlighted in the document** beside it. Correct anything wrong; mark a fact `MANUAL` to attest a value yourself; `NOT_FOUND` if it truly isn't there. When you're satisfied, **Confirm** — this **freezes** the fact set and classifies it in one step.
4. **Read the derivation** — the classification tab shows, per category, the impact band, likelihood, matrix rating, the tier your appetite demands, and which category **binds** the outcome; every escalation trigger (fired *and* not-fired — auditors need what was checked); and the plain-language explanation. If required facts were missing under a `needs_info` policy, you'll see **INCOMPLETE** with the missing list instead, and routing is blocked until you supply them.
5. **Approve** — the request routes to an approver's **Inbox**. They see the derivation, a due-time countdown, and the quorum progress, then approve (optional comment) or reject (comment required). A single rejection decides the task immediately; approvals accumulate to the quorum frozen at task creation. A tier-0 result auto-approves with no human step.
6. **Verify** — anyone with `auditor` can hit **Replay** on a classification: DDAS re-runs the stored facts through the pinned engine and confirms the derivation hash still matches. That replay *is* the audit.

## 5. AI agents (API keys + MCP)

Agents are first-class principals, gated exactly like people — same pipeline, same inbox, same audit trail — but held to the (usually stricter) agent appetite, and never allowed to self-approve.

1. Create an **agent** principal in **Admin → Principals** (it must name a human owner).
2. Mint a scoped **API key** for that agent (REST — the token is returned **once**):
   ```bash
   curl -X POST http://localhost:3000/api/v1/admin/api-keys \
     -H 'content-type: application/json' -b <admin-session-cookie> \
     -d '{"principalId":"<agent-id>","scopes":["requests:read","requests:write","facts:attest","mcp"]}'
   # → { "id": "...", "prefix": "...", "token": "ddas_<prefix>_<secret>" }  ← copy the token now
   ```
   Available scopes: `requests:read`, `requests:write`, `facts:attest`, `mcp`.

The agent then drives DDAS over **MCP** (streamable HTTP at `POST /mcp`, authenticated with the API key). Example client config:

```json
{
  "mcpServers": {
    "ddas": {
      "url": "http://localhost:3000/mcp",
      "headers": { "Authorization": "Bearer ddas_<prefix>_<secret>" }
    }
  }
}
```

Tools: `request_authority`, `get_decision_status`, `list_my_pending_requests`, `attest_fact`, `confirm_facts` — thin wrappers over the same domain logic as the REST API. The flow: the agent requests authority, reviews and attests facts (but **not** the human-gated ones — its owner does that), confirms, and then *waits* — it can never approve. A human approves in the inbox, and the agent's next status poll sees the green light. Every tool call is on the audit trail.

## 6. Integrations & operations

**Webhooks** (Admin → Webhooks) push audit events to your systems. Delivery rows are created in the same transaction as the event, so nothing is missed; deliveries are signed and retried, dead ones can be redelivered from the UI. Verify the signature:

```
X-DDAS-Signature: t=<unix-seconds>,v1=<hex hmac-sha256(secret, t + "." + rawBody)>
```

Reject anything where `|now − t| > 300s`; use `X-DDAS-Delivery` for idempotency. The signing secret is shown once when you register the webhook.

**SSO (OIDC)** — set `OIDC_ISSUER_URL`, `OIDC_CLIENT_ID`, `OIDC_CLIENT_SECRET`, `OIDC_REDIRECT_URL` (and optionally `OIDC_DEFAULT_ROLES`). A "Sign in with SSO" button appears; first login provisions the user just-in-time, linking to an existing account by email.

**Metrics** — Prometheus at `GET /metrics` (`ddas_*` counters). Dashboard queries: [docs/dashboards.md](docs/dashboards.md).

**Backup / restore** — `ddas backup create --out <dir>` writes a `pg_dump` + blob tarball + a manifest carrying the audit-chain head; `ddas backup restore --in <dir>` refuses a non-empty database and **fails loudly** if the restored chain doesn't reproduce that head.

**Tamper-evidence** — periodically `GET /api/v1/audit/checkpoint` and store the small JSON **outside** the deployment. Verifying against it later defeats even a database-level history rewrite.

**Kubernetes** — a single-replica, bring-your-own-Postgres Helm chart is in [`deploy/helm/ddas`](deploy/helm/ddas).

### Extraction providers

Set these to wire a real model (the only place DDAS talks to an LLM):

```bash
DDAS_EXTRACTION_PROVIDER=anthropic          # or: openai-compatible | stub
DDAS_EXTRACTION_MODEL=claude-sonnet-5
DDAS_EXTRACTION_API_KEY=...
# DDAS_EXTRACTION_BASE_URL=http://localhost:11434/v1   # openai-compatible (vLLM, Ollama, Azure…)
```

See [`.env.example`](.env.example) for the full set of environment variables.

Requires Node 24 and pnpm.

```bash
pnpm install
pnpm build
pnpm typecheck
pnpm test         # unit + integration; the db-backed suites need TEST_DATABASE_URL
```

Point the integration suites at a throwaway Postgres 16:

```bash
TEST_DATABASE_URL=postgres://postgres@localhost:5432/ddas_test pnpm test
```

The **golden corpus** (a fictional company, Kolvarra Industrial Systems) drives four things at once — extraction labels, engine snapshots, the HTTP end-to-end suite, and a demo seed:

```bash

node apps/cli/dist/main.js eval --engine     # replay labeled facts through the engine, offline
```

| Path | What |
|---|---|
| `packages/policy` | risk-policy schema, YAML→canonical-JSON compiler, linter, hashing |

## 8. Troubleshooting

- **Port 3000 already in use** — `DDAS_PORT=3210 docker compose up` (or change the published port in `deploy/docker-compose.yml`).
- **Submissions fail at extraction** — no extraction provider is configured. Set `DDAS_EXTRACTION_*`, or use `DDAS_EXTRACTION_PROVIDER=stub` for the manual-entry flow.
- **A request never reaches anyone / task shows "routing failed"** — there's no live approver at the required tier in the requester's org branch, so the task was assigned to admins. Add a position at that tier with a current assignee, or check your authority-ladder tier numbers match your positions.
- **Can't activate a policy** — lint findings of error severity block it; clear them first. The second and later activations want a simulation run (or a typed override reason).
- **Container logs** — `docker compose logs -f app`.
